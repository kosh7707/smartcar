# S5 ETL Pipeline Overview 문서 상세 피드백

검토 대상: `(WR)S5_etl_pipeline_overview.md`  
검토 목적: 외부 리뷰어·프로젝트 평가자 관점에서 문서의 **명확성, 기술적 일관성, 설득력, 운영성 설명, 평가 대응력**을 높이기 위한 상세 피드백

---

## 1. 총평

이 문서는 전반적으로 **문제 정의 → 데이터 소스 → ETL 단계 → 런타임 활용 → 설계 판단 → 코드 부록**의 흐름이 잘 잡혀 있고, 기술 문서로서의 밀도도 높다. 특히 다음 세 가지는 강점이다.

1. **왜 이 파이프라인이 필요한지**를 AEGIS 전체 분석 흐름 안에서 설명하고 있어, 단순 구현 설명서가 아니라 시스템 맥락 문서로 기능한다.
2. **CWE / ATT&CK / CAPEC의 역할 분담**을 비교표와 예시 경로로 설명해, 교차 참조의 필요성이 직관적으로 이해된다.
3. **설계 판단 근거**를 별도 섹션으로 정리하고, 일부는 수치까지 제시해 외부 평가자가 “왜 이렇게 설계했는가”를 추적하기 쉽다.

다만, 외부 리뷰어 기준으로 보면 현재 문서는 **아키텍처 개요서**, **운영 가이드**, **구현 부록**, **설계 의사결정 기록(ADR)**의 성격이 한 문서 안에 동시에 들어가 있다. 그 결과 문서의 장점이 많은 반면, 다음과 같은 핵심 약점도 분명하다.

- **문서 범위가 혼재**되어 있다. “개요(overview)”라기에는 구현 세부와 실행 절차가 많고, 반대로 “설계 명세”라기에는 일부 근거가 축약되어 있다.
- **가장 중요한 메시지 중 하나인 ‘CVE는 ETL에서 제외’**가 본문과 부록 코드 사이에서 일관되게 유지되지 않는다.
- **용어 체계가 완전히 정렬되어 있지 않다.** 예를 들어 ‘지식 그래프’, ‘Knowledge Base’, ‘하이브리드 검색’, ‘Qdrant 파일 DB’, ‘Neo4j 그래프 DB’가 함께 등장하는데, 전체 지식 저장소의 개념 모델이 정확히 무엇인지 한 문장으로 정리되어 있지 않다.
- **일부 수치와 기준값**(예: relevance threshold 0.2, 가중치 60/40, ATT&CK relevance 100%)은 제시되지만, 왜 타당한지에 대한 검증 또는 해석이 부족하다.

결론적으로, 이 문서는 **내용 자체는 좋지만, 외부 평가용 문서로서 한 단계 더 다듬을 여지가 매우 크다.** 특히 “현재 파이프라인의 정확한 범위와 제약”을 먼저 선명하게 만들고, 본문과 부록의 모델을 완전히 일치시키면 설득력이 크게 올라간다.

## 2. 가장 잘된 점

### 2.1 시스템 맥락 설명이 좋다

`2.1 AEGIS의 분석 흐름에서의 위치`는 매우 좋다. ETL이 단순 데이터 적재가 아니라, SAST 결과를 해석하기 위한 지식 기반의 선행 단계라는 점이 명확하다. 외부 리뷰어는 보통 “이 파이프라인이 제품 전체에서 왜 중요한가?”를 먼저 묻는데, 그 질문에 제대로 답하고 있다.

### 2.2 세 소스의 역할 분해가 설득력 있다

`2.2 세 가지 데이터 소스를 조합하는 이유`의 표는 강력하다. CWE는 취약점 분류, ATT&CK는 공격 기법, CAPEC은 공격 패턴 및 브릿지라는 구도가 잘 보인다. 여기에 이어지는 예시 경로(`CWE-787 -> CAPEC-100 -> T0831`)도 독자의 이해를 돕는다.

### 2.3 설계 근거가 “결론형”으로 정리되어 있다

`8. 설계 판단 요약`은 문서의 신뢰도를 높인다. 특히 다음 항목은 좋은 설계 설명이다.

- CVE를 ETL에서 제외한 이유
- 부모 체인 탐색으로 분류 성능이 개선된 이유
- 2,000건 규모에서 Qdrant 파일 기반 스토리지를 선택한 이유
- CAPEC을 브릿지 겸 풀 노드로 승격한 이유

외부 평가자는 대체로 “이 선택이 임의적이지 않았는가?”를 의심하는데, 이 섹션은 그 의심을 줄여 준다.

### 2.4 코드 부록이 기술적 신뢰를 보강한다

단순 설명에 그치지 않고 핵심 알고리즘을 보여 주는 점은 강점이다. 특히 RRF, crossref, taxonomy 예시는 문서가 실제 구현과 연결되어 있다는 인상을 준다.

## 3. 우선순위가 가장 높은 수정 사항

아래 항목들은 가능하면 **가장 먼저** 정리하는 것이 좋다. 현재 문서의 설득력을 가장 크게 떨어뜨릴 수 있는 지점들이다.

### 3.1 “CVE는 ETL에서 제외”와 부록 코드가 충돌한다

이 문서에서 가장 큰 문제다.

본문에서는 분명히 다음 메시지를 강하게 밀고 있다.

- 현재 ETL은 **CWE / ATT&CK / CAPEC** 중심이다.
- **CVE는 사전 적재하지 않고 런타임에 실시간 조회**한다.
- `POST /v1/cve/batch-lookup`으로 동적 제공한다.

그런데 부록의 `crossref.py` 예시는 다음 요소를 포함한다.

- `nvd_records`
- `cwe_to_cves`
- `related_cve`
- docstring의 `CWE<->CVE<->ATT&CK<->CAPEC`
- `UnifiedThreatRecord`의 `related_cve`
- `KnowledgeAssembler`의 `CVE-*` 추출

즉, 문서 전반의 핵심 메시지는 “CVE는 현재 ETL 범위에서 빠졌다”인데, 부록은 마치 **여전히 CVE가 ETL의 4번째 소스처럼 포함되는 구조**를 보여 준다. 외부 리뷰어 입장에서는 다음과 같은 의문이 생긴다.

- “현재 설계가 정말 CVE 제외가 맞는가?”
- “본문은 최신이고, 부록은 예전 코드인가?”
- “ETL 제외와 런타임 조회가 동시에 존재하는 하이브리드 구조인가?”
- “도대체 현재 기준의 canonical data model은 무엇인가?”

#### 권장 수정

다음 중 하나로 반드시 정리하는 것이 좋다.

**권장안 A — 부록을 현재 구조에 맞게 수정**
- ETL 개요 문서에서는 `nvd_records`, `related_cve`, `CWE<->CVE` 관련 코드를 제거하거나 별도 부록으로 분리한다.
- 현재 문서의 범위를 “사전 적재 파이프라인”으로 한정한다.

**권장안 B — 현재 구조가 ‘ETL + 런타임 enrichment’라면 그 사실을 명시**
- 본문 초반에 “정적 적재 범위는 3계열(CWE/ATT&CK/CAPEC)이고, CVE는 런타임 enrichment 계층에서 연결된다”고 명시한다.
- 부록 각 코드에 `[현재 ETL 범위]`, `[런타임 enrichment 포함 일반화 버전]` 같은 라벨을 붙인다.

#### 바로 적용 가능한 문장 예시

> 본 ETL 파이프라인의 사전 적재 범위는 CWE, ATT&CK, CAPEC 세 계열이다. CVE/NVD는 현재 정적 적재 대상에서 제외되며, 런타임 배치 조회 계층에서만 사용된다. 아래 일부 코드 예시는 향후 또는 런타임 enrichment까지 포함한 일반화된 자료구조를 보여 준다.

이 한 문장만 있어도 독자의 혼란이 크게 줄어든다.

### 3.2 문서 제목이 ‘Overview’인데 문서 내용은 운영 가이드와 구현 부록까지 포함한다

현재 제목은 `etl_pipeline_overview`인데, 실제 내용은 아래 네 성격이 섞여 있다.

1. **개요서**: 왜 필요한가, 전체 흐름은 무엇인가
2. **설계 문서**: 왜 Qdrant/Neo4j/CAPEC 브릿지를 선택했는가
3. **운영 가이드**: 실행 명령, 사전 조건, 경로, 캐시, 재빌드 옵션
4. **구현 부록**: 클래스, 함수, 알고리즘 코드

이 구성이 나쁜 것은 아니지만, 외부 리뷰어 관점에서는 메시지 밀도가 분산된다. 특히 `5. 실행 방법`, `7. 파일 구조`, `부록 A. 핵심 코드`는 정보 가치가 있지만, “왜 이 시스템 설계가 타당한가”를 보고 싶은 평가자에게는 상대적으로 비중이 커 보일 수 있다.

#### 권장 수정

문서를 둘로 나누는 방안을 강하게 권장한다.

- **문서 A: S5 ETL Pipeline Overview**
  - 문제 정의
  - 데이터 소스
  - 파이프라인 구조
  - 저장소 구조(Qdrant/Neo4j)
  - 설계 판단
  - 결과 및 한계
- **문서 B: S5 ETL Operator / Build Guide**
  - 실행 명령
  - 캐시 정책
  - 서비스 중지 조건
  - 디렉터리 구조
  - 장애 대응
  - 재빌드 절차
- **문서 C(선택): Implementation Notes / Appendix**
  - 코드 발췌
  - 스키마 상세
  - 의사결정 근거의 추가 설명

문서를 나누기 어렵다면, 최소한 본문에서 “운영 가이드”와 “구현 부록”을 뒤로 보내고, 초반에는 평가자가 알아야 할 핵심 메시지만 남기는 편이 좋다.

---

### 3.3 ‘지식 그래프’와 ‘하이브리드 지식 기반’의 개념이 섞여 있다

문서 초반은 `위협 지식 그래프`라고 소개하지만, 실제 저장 구조는 다음과 같다.

- **Qdrant**: 벡터 검색용 저장소
- **Neo4j**: 관계 탐색용 그래프 저장소

즉 시스템 전체는 엄밀히 말하면 **그래프 단일 저장소**가 아니라 **벡터 + 그래프를 결합한 하이브리드 지식 기반**에 가깝다. 그런데 첫 문장에서 “위협 지식 그래프”라고 쓰면, 외부 리뷰어는 “그럼 왜 Qdrant가 있는가?”라는 질문을 할 수 있다.

#### 권장 수정

첫 문장을 다음과 같이 바꾸는 것이 더 정확하다.

> AEGIS Knowledge Base(S5)는 자동차 임베디드 소프트웨어 취약점 분석을 위해, 벡터 검색(Qdrant)과 관계 탐색(Neo4j)을 결합한 하이브리드 위협 지식 기반을 운영한다.

이렇게 하면 이후 Qdrant와 Neo4j가 함께 등장해도 개념 충돌이 없다.

---

### 3.4 Neo4j가 “선택”인지 “사실상 필수”인지 분명하지 않다

`3.1`에서는 `Phase 5: [선택] Neo4j 시드`라고 쓰여 있다. 하지만 `6.1 하이브리드 검색`에서는 다음이 핵심 경로로 설명된다.

- ID 직접 조회 → Neo4j 조회
- 그래프 확장 → Neo4j 관계 보강

즉 문서만 읽으면, Neo4j는 “선택적 후처리”처럼 보이지만 실제 런타임 검색 설명에서는 상당히 중요한 역할을 한다.

#### 외부 리뷰어가 가질 수 있는 질문

- Neo4j가 없으면 서비스가 정상 동작하는가?
- 성능만 저하되는가, 기능이 비활성화되는가?
- 벡터 검색만으로 degraded mode가 있는가?
- `--seed`를 하지 않은 환경은 개발 모드인지, 제한된 운영 모드인지?

#### 권장 수정

문서에 아래 중 하나를 명시해야 한다.

- **선택 기능형**: “Neo4j가 없더라도 벡터 검색은 동작하며, 관계 확장만 비활성화된다.”
- **사실상 필수형**: “프로덕션 모드에서는 Neo4j 시드가 권장되며, 미시드 환경은 제한 모드이다.”

#### 바로 적용 가능한 문장 예시

> Neo4j 시드는 선택 단계이지만, 이를 수행하지 않은 경우 KB 서비스는 벡터 검색 중심의 축소 모드로 동작하며 그래프 이웃 확장 및 일부 ID 직접 관계 조회 기능은 제한된다.

이 한 문장이 있으면 문서의 운영 의미가 훨씬 명확해진다.

---

### 3.5 메인 스키마 설명과 부록 스키마 설명이 서로 다르다

`3.2 Phase 2`의 `UnifiedThreatRecord` 요약 스키마와 `A.6 schema.py`의 전체 스키마가 다르다.

본문 요약에는 다음이 보인다.

- `id`
- `source`
- `title`
- `description`
- `threat_category`
- `severity`
- `attack_surfaces`
- `automotive_relevance`
- `mitigations`
- `related_cwe`
- `related_attack`
- `related_capec`

부록 스키마에는 여기에 더해 다음이 추가된다.

- `attack_vector`
- `kill_chain_phase`
- `related_cve`
- `4-Layer` 구조 설명

그리고 카테고리 이름도 약간 달라진다.

- 본문: `Authentication`, `Configuration`
- 부록: `Authentication/Authorization`, `Configuration/Deployment`

이 차이는 개발자에게는 자연스러울 수 있지만, 외부 리뷰어는 이를 **문서 불일치**로 받아들일 가능성이 크다.

#### 권장 수정

- 본문에는 “단순화된 요약 스키마”라고 명시한다.
- 부록에는 “전체 구현 스키마”라고 명시한다.
- 용어를 완전히 통일한다.
- 현재 ETL 범위에서 사용하지 않는 필드는 `optional runtime-enrichment field` 등으로 표기한다.

## 4. 강하게 권장하는 중간 수준 개선 사항

### 4.1 “세 가지 데이터 소스”와 “4개 소스” 표현을 정리해야 한다

문서의 앞부분은 일관되게 **세 가지 데이터 소스(CWE, ATT&CK, CAPEC)**를 말한다. 그런데 `3.1 Phase 3`, `A.1 crossref.py`는 “4개 소스” 표현을 사용한다. 현재 맥락상 4번째 소스는 아마도 CVE/NVD이거나, ATT&CK ICS와 Enterprise를 분리 계산한 것처럼 읽힐 수 있다.

#### 권장 수정

다음처럼 표현 계층을 정리하면 좋다.

- **세 계열의 지식 원천**: CWE / ATT&CK / CAPEC
- **네 개의 입력 데이터셋**: CWE XML / ATT&CK ICS STIX / ATT&CK Enterprise STIX / CAPEC XML
- **CVE/NVD**: ETL 외부의 런타임 조회 계층

이렇게 정리하면 “세 가지 소스”와 “네 개의 입력 파일”이 동시에 참이 될 수 있다.

---

### 4.2 `severity` 필드 설명이 오해를 부를 수 있다

본문 스키마 설명에서는 `severity: float | None # CVSS 기반 심각도`라고 되어 있다. 하지만 CWE, ATT&CK, CAPEC는 일반적으로 CVSS 점수를 본원적으로 제공하는 데이터 소스가 아니다. 따라서 외부 리뷰어는 다음을 의문시할 수 있다.

- 이 필드는 실제로 어떤 소스에서 채워지는가?
- CVSS가 없는 MITRE 원천에 대해서는 자체 정규화 점수를 넣는가?
- `None`이 기본인가?
- source별 필드 의미가 달라지는가?

#### 권장 수정

이 필드는 설명을 더 엄밀하게 하는 것이 좋다.

예시:

> `severity`: 선택 필드. CVE/NVD와 같이 정량 점수가 있는 소스에서는 원점수 또는 정규화 점수를 저장할 수 있으며, CWE/CAPEC/ATT&CK와 같은 MITRE 지식 원천에는 기본적으로 비어 있을 수 있다.

이렇게 바꾸면 데이터 모델이 훨씬 현실적으로 보인다.

---

### 4.3 `automotive_relevance`라는 이름이 실제 범위를 충분히 설명하지 못한다

문서의 공격 표면 택소노미에는 아래 범주가 포함된다.

- CAN Bus, IVI, V2X, OTA, ECU, ADAS, 충전 인프라
- 임베디드/RTOS
- 시스템 라이브러리
- 산업제어/ICS

즉 이 점수는 순수한 “자동차성(automotive)”만이 아니라, **차량/임베디드/산업제어 인접성**까지 포함한 개념처럼 보인다. 그런데 필드 이름은 `automotive_relevance`이므로 범위가 더 좁게 읽힌다.

#### 권장 수정

다음 중 하나를 고려할 만하다.

- 필드명을 `domain_relevance`로 바꾸고, 문서에서 “automotive-embedded relevance”라고 설명
- 현재 이름을 유지하되, “자동차/임베디드/ICS 인접 도메인 관련성 점수”라고 명시

현재처럼 두면 ATT&CK relevance 100% 같은 수치가 더 의아하게 보일 수 있다.

---

### 4.4 ATT&CK relevance 100%는 설명이 없으면 과장처럼 보일 수 있다

현재 상태 표에는 다음이 나온다.

- CWE 약 48%
- ATT&CK 100%
- CAPEC 약 35%

이 숫자 자체가 틀렸다는 뜻은 아니다. 다만 외부 리뷰어는 다음을 물을 가능성이 높다.

- 왜 Enterprise ATT&CK까지 포함하는데 100%인가?
- relevance threshold가 너무 낮은 것 아닌가?
- ATT&CK는 파싱 단계에서 이미 필터링된 것인가?
- automotive relevance가 아니라 “보안 분석 관련성”에 가까운 것 아닌가?

#### 권장 수정

숫자를 유지하더라도 해석을 붙이는 것이 좋다.

예시:

> ATT&CK 레코드는 순수한 자동차 전용 기법만 의미하지 않는다. 본 점수는 차량/임베디드 분석에 유용한 보안 행위 지식까지 포함하는 operational relevance로 정의했기 때문에, 현재 수집 대상 ATT&CK 집합은 전량 관련 항목으로 간주된다.

또는,

> ATT&CK는 별도 relevance 필터를 통과한 subset만 적재하므로 결과적으로 100%를 보인다.

이 설명이 없으면 데이터 선별 기준이 불명확해 보인다.

---

### 4.5 relevance threshold 0.2, 키워드 60% + 임베딩 40%의 근거를 보강해야 한다

현재 문서는 방법은 설명하지만, 기준값의 정당성은 충분히 설명하지 않는다.

외부 리뷰어는 다음을 묻기 쉽다.

- 왜 threshold가 0.2인가?
- 왜 60/40인가?
- 검증셋이 있었는가?
- false positive / false negative의 경향은 어떠한가?
- 키워드 사전 63개는 어떻게 선정했는가?

#### 권장 수정

정량 평가가 완전하지 않더라도 아래 수준의 설명은 추가하는 것이 좋다.

- 소규모 샘플 수동 검증 결과
- 몇 개의 대표 사례에서 threshold를 정한 이유
- 60/40이 경험적 휴리스틱임을 명시
- 향후 개선 계획(예: labeled validation set 구축)

예시:

> 현재 relevance 가중치와 threshold는 소규모 수동 검토를 통해 정한 경험적 휴리스틱이다. 향후에는 labeled validation set을 구축해 precision/recall 관점에서 재조정할 예정이다.

이 문장이 있으면 “임의 값”이라는 인상을 줄일 수 있다.

---

### 4.6 공격 표면 분류 체계는 유용하지만, 분류 축이 혼합되어 있다

현재 11개 분류는 실무적으로는 유용할 수 있지만, 택소노미 관점에서는 다음 축이 한 표에 섞여 있다.

- **차량 인터페이스/채널**: CAN Bus, V2X, OTA, 충전 인프라
- **차량 서브시스템**: IVI, ECU, ADAS, 키/인증 시스템
- **소프트웨어/플랫폼 계층**: 임베디드/RTOS, 시스템 라이브러리
- **인접 산업 도메인**: 산업제어/ICS

즉 이것은 하나의 직교적인 taxonomy라기보다, **운영상 유용한 태그 묶음**에 가깝다. 현재 문서는 이를 “11개 택소노미”라고 부르기 때문에, 엄밀한 분류 체계처럼 읽히는 문제가 있다.

#### 권장 수정

아래 중 하나를 권장한다.

- “11개 공격 표면 taxonomy” 대신 “11개 운영 태그 세트”라고 완화
- 분류를 2단계 구조로 변경
  - 1차: 채널 / 서브시스템 / 플랫폼 / 인접도메인
  - 2차: 세부 태그

최소한 문서에는 다음 정도의 주석이 있으면 좋다.

> 본 분류는 학술적 직교 분류체계라기보다, 차량 보안 분석 시 검색 및 필터링 효율을 높이기 위한 실용적 태그 세트이다.

---

### 4.7 “부모 체인 5단계 탐색”의 개선 효과는 좋지만, 아직 `Other` 52%라는 사실도 함께 해석해야 한다

`Other 89% -> 52%` 개선은 매우 좋은 근거다. 다만 외부 리뷰어는 자연스럽게 다음 질문을 한다.

- 52%면 여전히 절반 이상이 Other인데, 이 분류가 충분히 유용한가?
- 이 8개 상위 카테고리 자체가 너무 거친 것 아닌가?
- 향후 수작업 큐레이션 또는 다중 카테고리 분류 계획이 있는가?

#### 권장 수정

현재 수치가 나쁜 것은 아니지만, 다음 해석을 붙이면 훨씬 성숙해 보인다.

> 부모 체인 탐색 도입으로 대분류 분해 성능은 유의미하게 개선되었으나, 여전히 많은 CWE가 일반 상위 개념 또는 애매한 메타 수준에 머문다. 따라서 현재 카테고리는 정밀 분류기라기보다 검색 보조용 coarse tag로 사용하는 것이 적절하다.

이 설명은 오히려 문서를 더 신뢰할 수 있게 만든다.

---

### 4.8 CAPEC 버전과 신선도에 대한 설명이 있으면 좋다

문서에는 `CAPEC v3.9, 2023-01-24`가 적혀 있다. 문서 작성일은 2026-03-26이므로, 외부 리뷰어는 자연스럽게 “왜 CAPEC이 오래된 것처럼 보이는가?”를 물을 수 있다.

실제로 최신 여부가 무엇이든, 문서가 해야 할 일은 독자의 의문을 줄이는 것이다.

#### 권장 수정

다음 중 하나를 명시하면 좋다.

- “해당 버전은 빌드 시점 기준 최신 공개 버전이다.”
- “상위 원천의 릴리스 주기가 느려 해당 날짜가 오래되어 보이지만 정상이다.”
- “호환성/재현성을 위해 특정 버전을 핀했다.”

이 설명이 없으면 단순히 데이터가 방치된 것처럼 보일 위험이 있다.

---

### 4.9 ATT&CK ICS와 Enterprise 병합 전략을 조금 더 설명하면 좋다

문서에서는 `ATT&CK v18.1, ICS 83 + Enterprise 426 = 509`라고 한다. 숫자는 깔끔하지만, 외부 리뷰어는 다음을 궁금해할 수 있다.

- 두 데이터셋 간 중복 ATT&CK ID는 없는가?
- 중복이 있다면 merge 기준은 무엇인가?
- sub-technique는 포함되는가?
- vehicle domain relevance 관점에서 Enterprise technique를 왜 유지하는가?

간단히라도 병합 원칙을 설명하면 문서의 완성도가 올라간다.

예시:

> ATT&CK는 ICS와 Enterprise를 통합하되, 동일 ATT&CK ID 기준으로 dedup한다. 차량 보안 맥락에서는 순수 OT 계열뿐 아니라 일반 소프트웨어 공격 기법도 유의미하므로 Enterprise 기술도 함께 유지한다.

---

## 5. 외부 평가 관점에서 추가되면 좋은 내용

### 5.1 재현성(reproducibility) 설명 강화

현재 캐시 경로와 `kb-meta.json` 설명은 좋다. 하지만 외부 평가자 관점에서는 재현성 질문이 더 나올 수 있다.

권장 보강 항목:

- ATT&CK GitHub 소스의 **commit hash 또는 release tag** 기록
- 다운로드 파일의 checksum 또는 최소한 version stamp 기록
- 빌드 날짜뿐 아니라 **source snapshot 날짜** 명시
- 캐시 hit/miss 여부가 결과에 영향을 주는지 명시

현재 문서는 “버전을 기록한다”까지는 설명하지만, “같은 결과를 다시 만들 수 있는가?”에 대한 확신은 조금 약하다.

### 5.2 실패 모드와 예외 처리 설명

현재 문서에는 정상 경로는 잘 설명되어 있지만, 실패 시 행동이 없다.

외부 리뷰어는 보통 이런 질문을 한다.

- 다운로드 실패 시 전체 빌드를 중단하는가, 부분 빌드를 허용하는가?
- ATT&CK만 내려받고 CAPEC 다운로드에 실패하면 어떻게 되는가?
- 스키마 변경으로 XML/JSON 파싱이 실패하면 버전 체크를 하는가?
- 이전 캐시가 있으면 fallback하는가?

아주 길게 쓸 필요는 없고, `Known constraints / failure modes` 소절 하나면 충분하다.

### 5.3 품질 평가(quality evaluation) 또는 검증 절차

현재 문서는 파이프라인 구조는 잘 설명하지만, “결과가 얼마나 쓸 만한가”에 대한 검증 설명은 적다. 외부 평가자가 보기에 가장 좋은 보강은 아래 중 하나다.

- crossref 품질 샘플 검토
- relevance score 수동 검증 샘플
- 하이브리드 검색 결과의 정성 평가
- false positive 사례 / known limitations

정량 지표가 완전하지 않아도, 최소한 `검증 방법`을 짧게 소개하면 문서 신뢰도가 올라간다.

### 5.4 보안/무결성 관점 언급

이 문서는 보안 프로젝트 문서이므로, 외부 원천 데이터를 다운로드하는 과정의 무결성이나 신뢰 경계도 한 줄 정도 언급되면 좋다.

예시:

- 외부 소스는 공식 MITRE / 공식 GitHub repository에 한정
- 버전 및 메타데이터 기록
- 향후 checksum 검증 또는 schema validation 강화 예정

이 내용은 문서의 전문성을 높여 준다.

## 6. 섹션별 상세 피드백

## 6.1 `1. ETL 파이프라인이란?`

### 좋은 점
- 첫 단락이 간결하다.
- 입력 소스 → ETL → 저장소라는 그림이 직관적이다.

### 보완점
- “위협 지식 그래프” 대신 “하이브리드 위협 지식 기반”이 더 정확하다.
- 이 문단에서 문서 범위를 한 줄로 못 박는 것이 좋다.

#### 권장 보강 문장

> 본 문서는 사전 적재 대상(CWE/ATT&CK/CAPEC)과 그 변환·적재 흐름을 설명하며, CVE/NVD의 런타임 조회는 범위 밖으로 둔다.

이 문장이 초반에 있으면 이후 혼선이 줄어든다.

---

## 6.2 `2. 왜 필요한가?`

### 좋은 점
- AEGIS 내부 위치 설명이 매우 좋다.
- 세 소스 조합 이유 설명도 강하다.

### 보완점
- S3, S4, S5라는 이름은 내부자에게 익숙하지만 외부 리뷰어에겐 낯설 수 있다.
- 각 약어의 풀네임 또는 한 줄 정의를 붙이면 더 친절하다.

#### 권장 수정

예시:

- S4 (Static Analysis Runner)
- S5 (Knowledge Base)
- S3 (Analysis Agent)

이미 일부는 써 두었지만, 첫 등장 시 더 명시적으로 적어도 좋다.

---

## 6.3 `2.3 CVE는 왜 ETL에서 제외되었는가?`

### 좋은 점
- 문서 내에서 가장 설득력 있는 비교표 중 하나다.
- “사전 적재보다 런타임 조회가 더 정확하고 효율적”이라는 메시지가 잘 전달된다.

### 보완점
- endpoint 이름까지는 운영/구현 정보에 가까워서 overview 문서에서는 다소 세부적일 수 있다.
- 대신 아키텍처 관점의 메시지를 더 강조하는 편이 좋다.

#### 권장 수정 방향

현재 문장:
> 현재 `POST /v1/cve/batch-lookup` 엔드포인트로 실시간 제공한다.

대안:
> 현재 CVE 정보는 별도 런타임 조회 계층에서 제공되며, 프로젝트의 실제 라이브러리 및 버전 기준으로 동적으로 결합된다.

API 문서가 따로 있다면 endpoint 세부명은 그쪽으로 빼는 편이 낫다.

---

## 6.4 `3. 파이프라인 구조`

### 좋은 점
- 5단계 구조가 깔끔하다.
- 각 Phase가 무엇을 하는지 대체로 잘 보인다.

### 보완점
- `Crossref`가 정확히 언제 어떤 소스에 대해 작동하는지 명확히 정리하면 좋다.
- `4개 소스` 표현은 수정이 필요하다.

### 추가로 있으면 좋은 것
- 각 단계의 **입력/출력 형태**를 한 줄씩 명시
  - Download: raw XML/JSON files
  - Parse: normalized records + bridge tables
  - Crossref: enriched normalized records
  - Load: vector collection + metadata
  - Stats: build summary + version metadata

이렇게 쓰면 파이프라인 문서로서 훨씬 또렷해진다.

---

## 6.5 `3.2 Phase 2: 데이터 파싱`

### 좋은 점
- `UnifiedThreatRecord`를 초반에 보여 주는 선택은 좋다.
- 도메인 관련성 / 카테고리 분류 / 공격 표면 분류를 한데 묶어 설명한 것도 자연스럽다.

### 보완점
- 현재 설명은 “정규화”와 “추론/분류”가 한 단계에 섞여 있다.
- 엄밀하게는 다음을 분리해 쓰는 편이 좋다.
  - source parsing / schema normalization
  - enrichment (category, surface, relevance)

즉 “Parse”가 너무 많은 일을 하는 것처럼 보인다. 외부 리뷰어에게는 “파싱”과 “도메인 추론”은 다른 수준의 작업으로 읽힌다.

#### 권장 표현

> Phase 2는 원천 데이터를 공통 스키마로 정규화하고, 이어지는 enrichment 단계에서 도메인 관련성·위협 카테고리·공격 표면 태깅을 수행한다.

---

## 6.6 `3.2 도메인 관련성 점수`

### 좋은 점
- 가중치 구조가 공개되어 있어 투명하다.
- 키워드와 임베딩의 상호 보완 논리가 이해된다.

### 보완점
- 왜 키워드 60 / 임베딩 40인지 근거가 없다.
- 키워드 리스트에 자동차, 임베디드, 시스템, ICS가 함께 들어가므로 `automotive_relevance`라는 이름과 다소 어긋난다.
- `_RELEVANCE_REF`처럼 하나의 기준 문장과의 cosine similarity만으로 충분한가에 대한 설명이 없다.

#### 권장 보강

- “경험적 휴리스틱”임을 명시
- labeled set 부재 또는 향후 계획을 밝힘
- field 명칭의 의미 범위를 분명히 함

## 6.15 `부록 A`

### 좋은 점
- 구현 신뢰도를 높인다.
- RRF 예시는 특히 좋다.

### 핵심 보완점
1. 현재 ETL 범위와 직접 연결되지 않는 CVE 관련 코드는 정리 필요
2. 본문 스키마와 부록 스키마의 차이를 명시 필요
3. `4-Layer` 개념은 본문 어디에도 없으므로 갑자기 등장하면 혼란스럽다
4. 카테고리 명칭을 본문과 통일해야 한다
5. 실행 가능한 코드가 아니라 “발췌”라는 점을 더 명확히 해야 한다

#### 권장 문구

> 아래 코드는 설명 목적의 발췌본이며, 일부는 현재 ETL 범위를 넘어 런타임 enrichment 또는 일반화된 자료구조를 포함한다.

이 문구가 있으면 부록의 자유도가 커진다.

## 7. 표현과 용어 차원의 교정 포인트

### 7.1 용어 일관성 표

| 현재 표현 | 문제 | 권장 방향 |
|---|---|---|
| 위협 지식 그래프 | Qdrant까지 포함한 전체 구조를 충분히 설명하지 못함 | 하이브리드 위협 지식 기반 |
| 세 가지 데이터 소스 | 뒤에서 4개 소스 표현과 충돌 가능 | 세 계열의 지식 원천 / 네 개의 입력 데이터셋 |
| automotive_relevance | 실제 범위가 자동차 + 임베디드 + ICS 인접성까지 포함 | domain_relevance 또는 의미 확장 설명 |
| 공격 표면 taxonomy | 일부 항목은 표면이 아니라 플랫폼/환경 태그 | 공격 표면 및 플랫폼 태그 세트 |
| severity (CVSS 기반) | MITRE 원천에 일반적으로 직접 대응되지 않음 | 선택 필드 / source-dependent score |
| 선택적 Neo4j 시드 | 런타임 설명에서는 중요해 보임 | 축소 모드/제한 모드 여부 명시 |

### 7.2 문장 톤 조정

현재 문장은 전반적으로 좋지만, 몇몇 문장은 “단정”이 강해 보일 수 있다. 외부 평가용 문서에서는 다음처럼 쓰면 더 안정적이다.

- “유일한 공식 경로” → “현재 채택한 공식 연결 경로” 또는 “현재 문서 기준의 공식 연결 경로”
- “부적합” → “현재 실험 범위에서는 적합성이 낮았다”
- “전부 관련” → “현재 기준에선 관련 집합으로 간주”

이런 조정은 문서를 덜 방어적으로 만들고, 질문을 받았을 때도 답변이 쉬워진다.

## 10. 바로 반영하기 좋은 수정 예시 문안

### 예시 1. 문서 첫 문단 교체안

> AEGIS Knowledge Base(S5)는 자동차 임베디드 소프트웨어의 보안 취약점 분석을 위해, 벡터 검색(Qdrant)과 관계 탐색(Neo4j)을 결합한 하이브리드 위협 지식 기반을 운영한다. 본 문서가 설명하는 ETL 파이프라인은 CWE, ATT&CK, CAPEC 세 계열의 공개 위협 지식을 수집·정규화·교차참조·적재하는 오프라인 빌드 과정이며, CVE/NVD는 현재 정적 적재 범위에서 제외하고 런타임 조회 계층에서 다룬다.

이 문단 하나만으로도 범위, 저장 구조, CVE 제외 원칙이 동시에 정리된다.

### 예시 2. Neo4j 선택성 설명 보강안

> Neo4j 시드는 선택 단계이지만, 미시드 환경에서는 KB 서비스가 벡터 검색 중심의 축소 모드로 동작한다. 이 경우 그래프 이웃 확장과 일부 관계 기반 조회 기능은 제한된다.

### 예시 3. relevance 설명 보강안

> `automotive_relevance`는 순수한 자동차 전용성만이 아니라 차량·임베디드·인접 제어 도메인에서의 분석 유용성을 함께 반영한 운영 지표다. 현재 threshold와 가중치는 소규모 수동 검토를 바탕으로 정한 경험적 휴리스틱이며, 향후 라벨드 검증셋으로 재조정할 계획이다.

### 예시 4. 공격 표면 분류 설명 보강안

> 본 11개 분류는 엄밀한 학술 taxonomy라기보다, 차량 보안 분석에서 검색·필터링 효율을 높이기 위한 실용적 태그 세트이다. 일부 항목은 네트워크 표면, 일부는 차량 서브시스템, 일부는 플랫폼/라이브러리 계층을 나타낸다.

## 12. 한 줄 결론

현재 문서는 **기술적 내용과 설계 논리는 충분히 강하다.** 문제는 내용의 부족이 아니라 **범위 정의와 표현 일관성**이다. 특히 **“현재 ETL은 무엇을 포함하고, 무엇을 포함하지 않는가”**를 본문과 부록 전체에서 하나의 모델로 정렬하면, 외부 리뷰 대응력과 문서 완성도가 크게 올라간다.
