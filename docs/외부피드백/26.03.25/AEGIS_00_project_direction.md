# AEGIS가 나아갈 방향

## 1. 진단

AEGIS는 “자동차 임베디드 보안 검증용 분석 플랫폼”으로서의 정체성이 이미 분명하다.  
공개 저장소만 봐도 이 프로젝트는 단순 스캐너 집합이 아니라 다음 네 가지를 동시에 가진다.

- 계약 중심의 다중 서비스 구조
- 결정론적 분석 결과를 우선하는 보안 파이프라인
- LLM을 직접 판단 주체가 아니라 제한된 보조 계층으로 두는 안전한 운영 철학
- 분석가가 근거를 검토하고 승인/보류/반박할 수 있는 워크플로

즉, 현재 부족한 것은 “좋은 아이디어”가 아니라 **마감 품질과 운영적 정제**다.  
10일 내외의 구현치고는 구조가 이미 상당히 높다. 지금부터의 방향은 새 기능을 무작정 늘리는 것이 아니라, 이 구조를 오래 버티는 플랫폼으로 다듬는 데 맞춰야 한다.

---

## 2. 절대로 잃지 말아야 할 것

### 2.1 계약서로만 소통하는 팀 구조

이 프로젝트의 가장 강한 차별점은 기술 스택이 아니라 개발 방식이다.  
S1~S7이 서로의 코드를 보지 않고, API 계약과 명세로만 연결된다는 개념은 단순한 팀 운영 철학이 아니라 저장소 구조에도 반영되어 있다.

이 방식은 계속 유지하는 편이 좋다. 이유는 세 가지다.

- 서비스 경계가 억지로라도 유지된다.
- “편해서 직접 읽고 맞춘다”는 습관이 사라져 문서 품질이 강제된다.
- 이후 인원 교체, 병렬 개발, 외부 협업, 산출물 평가에서 설명력이 매우 높다.

다만 이 원칙은 이제 한 단계 더 가야 한다.  
앞으로는 “문서로 소통한다”를 넘어서 **문서와 테스트가 자동으로 계약 위반을 감지하는 상태**로 발전해야 한다.

권장 방향:
- `docs/api`를 단순 설명서가 아니라 버전 관리되는 계약 원본으로 격상
- consumer-driven contract test 도입
- 주요 요청/응답 예시를 golden fixture로 고정
- 공유 모델 변경 시 영향 서비스 목록과 마이그레이션 규칙을 함께 기록

---

## 3. 현재 구조를 어떻게 이해해야 하는가

현재 AEGIS의 구조는 크게 다음처럼 읽힌다.

- **S2**: 시스템의 상태와 실행 흐름을 관리하는 플랫폼 커널
- **S4**: 빠르고 재현 가능한 결정론적 정적 분석 엔진
- **S5**: 코드 그래프, 위협 지식, 프로젝트 메모리를 결합하는 기억 계층
- **S7**: LLM을 안전하게 사용하는 정책 게이트웨이
- **S3**: S4/S5/S7을 조합해 더 깊은 분석을 수행하는 bounded agent
- **S1**: 분석가가 실제로 결과를 이해하고 조치하도록 만드는 제품 표면
- **S6**: 미래의 동적 분석/ECU 검증 확장을 위한 프로토콜 계층

이 구조는 매우 좋다.  
왜냐하면 “LLM이 주인공인 보안 프로젝트”로 흐르지 않고, **결정론적 증거 → 오케스트레이션 → 제한적 추론 → 인간 검토**라는 순서를 유지하고 있기 때문이다.

앞으로의 방향도 이 순서를 강화하는 쪽이어야 한다.

---

## 4. 다음 단계의 핵심 과제

### 4.1 기능 추가보다 인터페이스 안정화가 먼저다

현재는 각 서비스의 역할이 이미 충분히 많다.  
추가 기능을 늘리기 전에 먼저 아래를 안정화해야 한다.

- 상태 전이 모델
- 이벤트/웹소켓 메시지 형식
- 공통 에러 포맷
- requestId 전파
- evidenceRef, finding, run, gateResult 같은 핵심 개체의 의미
- build target 및 subproject 단위 파이프라인 정의

특히 AEGIS에서 중요한 것은 “취약점 검출률”만이 아니라, **어떤 증거를 어떤 단계에서 누가 생성하고 승인했는지**를 추적할 수 있는 일관성이다.  
따라서 향후 방향은 기능 확장보다도 “플랫폼 언어”를 안정화하는 것이다.

### 4.2 S2를 더 강하게 만들되, 더 비대하게 만들면 안 된다

현재 구조상 S2가 실질적인 시스템 중심이다. 이것 자체는 맞다.  
문제는 시간이 지나면 S2가 두 종류의 코드를 모두 떠안기 쉽다는 점이다.

- 정말로 중앙에 있어야 하는 코드
- 원래는 서비스 밖으로 빼야 할 조립/부트스트랩/통합 편의 코드

앞으로의 방향:
- S2는 **상태, 실행 흐름, 결과 정규화, 승인/보고 책임**에 집중
- 부트스트랩, 라우트 조립, 외부 클라이언트 생성, 구성 결합은 모듈화
- 웹소켓 이벤트와 파이프라인 진행 상태는 공유 계약으로 승격
- 장기적으로는 “오케스트레이터”와 “애플리케이션 API 레이어”의 경계를 더 분명히 할 것

즉, S2는 더 중요해져야 하지만 더 무거워져서는 안 된다.

### 4.3 S4/S5를 더 깊게 만들고, S7은 더 엄격하게 제한해야 한다

AEGIS의 경쟁력은 결국 S4와 S5의 품질에 달려 있다.

- S4가 더 좋은 정적 근거를 만든다.
- S5가 그 근거를 지식과 연결한다.
- S7은 그 위에서만 제한적으로 추론한다.

이 질서를 지키면 프로젝트는 오래 간다.  
반대로 S7이 직접 많은 것을 “생성”하기 시작하면 프로젝트의 신뢰성이 빨리 무너질 수 있다.

따라서 방향은 다음과 같다.

- S4: 더 많은 규칙, 더 좋은 빌드/SDK 처리, 더 나은 code graph, 더 좋은 benchmark
- S5: 더 명확한 ontology, 더 신뢰도 높은 memory/graph merge, 더 안정적인 index lifecycle
- S7: task allowlist 유지, evidence whitelist 강제 유지, prompt/model/version 추적 강화

## 5. 30일 우선순위

### 5.1 저장소 정리와 공학적 마감

가장 먼저 할 일은 화려한 기능이 아니라 저장소를 믿을 수 있게 만드는 것이다.

반드시 권장하는 항목:
- 커밋된 바이너리/오브젝트 파일 제거
- stray file 정리
- `env.example` 정비
- 내부 IP, 사설 엔드포인트, 운영 세부정보를 공개 문서에서 분리
- CORS 기본 허용 범위 축소
- `SECURITY.md` 작성
- 설치 방식 일관화: workspace에 포함할 것과 별도 설치할 것을 명확히 분리
- 서비스별 실행/테스트 명령을 루트에서 통일적으로 호출 가능하게 구성

이 단계는 보기 좋은 정리가 아니라, **프로젝트 신뢰도와 재현성의 핵심**이다.

### 5.2 CI를 전면 도입

현재 수준의 멀티서비스 구조는 사람의 기억으로 유지할 수 없다.

최소 CI 세트:
- frontend test
- backend unit/integration/contract test
- analysis-agent test
- build-agent test
- sast-runner test
- knowledge-base test
- llm-gateway test
- markdown lint / API contract drift check

추가 권장:
- `docs/api` 스냅샷 검증
- golden request/response example 검증
- OpenAPI 또는 JSON schema 자동 산출
- PR 단위 서비스 영향 범위 표시

### 5.3 공통 이벤트 모델 정리

현재 AEGIS는 run, finding, evidence, quality gate, pipeline progress, websocket notification이 중요한 시스템이다.  
따라서 다음 개체는 가능한 한 빨리 공유 모델로 승격시키는 편이 좋다.

- RunStage
- PipelineStage
- ProgressEvent
- FindingDisposition
- EvidenceRef kind/source
- GateDecision
- ApprovalAction
- AgentClaim/AgentCaveat

이 작업을 먼저 해두면 S1, S2, S3, S7 사이의 의미 충돌이 크게 줄어든다.

## 6. 60일 우선순위

### 6.1 “연구용 데모”에서 “지속 가능한 연구 플랫폼”으로

이 시점에는 서비스별 품질 차이를 줄여야 한다.  
현재 공개 저장소 기준으로는 서비스별 완성도 편차가 존재한다.

- 매우 강한 편: S2, S4, S5, S7, 분석 에이전트
- 성장 여지가 큰 편: build-agent, S6, S1의 QA 체계화

따라서 60일 목표는 “모든 서비스가 같은 깊이를 갖게 하는 것”이 아니라, **핵심 경로를 따라 weakest link를 줄이는 것**이다.

권장 순서:
1. S2 이벤트/상태 모델 고정
2. S1 라우트/화면/QA 플로우 정렬
3. build-agent 실전 테스트 보강
4. S4 benchmark/fixture 강화
5. S5 graph/schema/memory lifecycle 문서화
6. S7 평가셋과 운영 제한 강화

### 6.2 제품 경험 정리

분석 플랫폼이 실제로 설득력을 가지려면 “검출했다”보다 “검토하고 결론 내리기 쉬운가”가 중요하다.

따라서 S1/S2 중심으로 다음이 필요하다.

- finding triage workflow의 일관화
- report 생성과 approval 흐름의 시각적 완결성
- 재분석 시 diff 보기
- build target / subproject / run 간 관계를 한눈에 보여주는 정보 구조
- 동적 분석이 미완이라도 “현재 범위”와 “예정 범위”를 UI에서 명확히 표현

## 7. 90일 우선순위

### 7.1 동적 분석(S6)의 다음 한 단계만 명확히 정의

S6는 지금 상태가 오히려 좋다.  
왜냐하면 현재 프로젝트 중심축이 정적 분석이므로, S6를 과도하게 키우지 않고 작고 교체 가능한 프로토콜 계층으로 두었기 때문이다.

90일 시점의 권장 목표는 “복잡한 시뮬레이터를 만드는 것”이 아니다.  
대신 아래 중 하나를 명확히 달성하는 편이 좋다.

- deterministic CAN replay
- scenario scripting DSL
- capture → replay → anomaly comparison
- one-click backend-driven dynamic test session

핵심은 화려함이 아니라 **재현성**이다.

### 7.2 평가 체계 고도화

프로젝트가 다음 단계로 가려면 서비스별 “좋아 보이는 구현”을 넘어서 다음이 필요하다.

- S4 benchmark pack
- S7 task-type별 golden evaluation
- S3 deep-analyze 품질 평가 기준
- false positive / accepted risk / resolved memory의 추적 지표
- run-to-run consistency metric

이렇게 되면 AEGIS는 단순 구현물이 아니라 **측정 가능한 연구 플랫폼**이 된다.

## 8. 서비스별 방향 요약

### S1
- UI를 더 많이 만들기보다, 이미 있는 정적 분석/리포트/승인 흐름을 더 탄탄히
- dynamic page와 실제 route 등록 상태를 일치시킬 것
- QA를 문서가 아닌 테스트 자산으로 승격

### S2
- source of truth 역할 유지
- 다만 composition root, bootstrap, orchestration을 분리해 god service화를 막을 것
- run/finding/evidence/gate 모델을 더 명시적으로 고정

### S3
- analysis-agent는 현재 방향 유지
- build-agent는 테스트와 build provenance를 대폭 보강
- 두 에이전트가 공유하는 정책/예산/도구 실행 커널은 정리할 가치가 큼

### S4
- 더 강해질 여지가 가장 큰 결정론적 코어
- benchmark, fixture, reproducibility, toolchain packaging이 중요
- scanner 집합이 아니라 “정적 분석 플랫폼”으로 승격할 수 있음

### S5
- knowledge, code graph, project memory의 세 영역을 명확히 구분
- ontology versioning과 memory lifecycle 정책이 다음 핵심
- AEGIS 전체 설명력을 끌어올리는 레버리지 서비스

### S6
- 작게 유지하는 것이 맞음
- 프로토콜 검증과 replay 기능부터
- 현재는 과투자 금지

### S7
- 지금처럼 gateway를 정책 장치로 유지
- LLM을 더 자유롭게 쓰는 것이 아니라 더 추적 가능하게 써야 함
- prompt/model/task/evidence 버전 관리가 핵심

## 9. 일부러 늦춰도 되는 것

다음은 지금 당장 하지 않아도 된다.

- 복잡한 서비스 메시/분산 인프라
- 외부 사용자 대상의 무거운 인증 체계
- 지나치게 많은 동적 분석 시나리오
- 자동 패치 생성
- 과도한 UI 장식
- premature multi-tenant 설계

현재 AEGIS에 필요한 것은 “대형 제품의 흉내”가 아니라, **핵심 보안 분석 경로의 완성도**다.

## 11. 결론

AEGIS가 나아갈 방향은 명확하다.

이 프로젝트는 이미 “무엇을 만들지”에 대한 답을 갖고 있다.  
이제 필요한 것은 아래 세 가지다.

1. **서비스 경계를 더 계약적으로 만든다.**
2. **결정론적 근거 계층(S4/S5)을 더 깊게 만든다.**
3. **오케스트레이션과 사용자 경험(S2/S1)을 더 안정화한다.**

즉, 앞으로의 방향은 “LLM 기능을 더 넣는 것”이 아니라  
**근거, 상태, 계약, 재현성을 더 강하게 만드는 것**이다.

그 방향만 유지하면 AEGIS는 짧은 기간에 만든 화려한 데모가 아니라,  
실제로 오래 설명되고 확장될 수 있는 보안 분석 플랫폼으로 갈 수 있다.
