# S4 → S3: SAST 도구 통합 및 GraphRAG 로드맵 제안

> 작성: S4 (2026-03-17)

---

## 배경

S4에서 모델 서빙이 안정화되면서(배치 7건 전부 200 OK, KV cache 2.1%), 다음 단계로 **분석 품질 고도화**를 논의했습니다. 현재 LLM이 코드를 맨눈으로 보고 취약점을 판단하는 구조는 근거(Evidence)가 부족하고, 에이전트 아키텍처로 가기 전에 기반을 다져야 합니다.

---

## 제안 1: SAST 도구 통합 (우선순위 높음)

### 현재 문제

- LLM이 코드 텍스트만 보고 취약점을 "추측"하는 구조
- 구조화된 증거(어떤 함수, 몇 번째 줄, 어떤 데이터 흐름) 없이 판단
- False positive 검증 수단 부재

### 제안하는 흐름 변화

```
[현재]
S2: 코드 청크 → S3: 코드 + RAG → S4: 추론
LLM 역할: 탐지자 (코드에서 취약점 찾기)

[SAST 통합 후]
S2: 코드 → SAST 도구 실행 → findings + 코드 → S3: findings + 코드 + RAG → S4: 추론
LLM 역할: 해석자 (SAST 결과 검증, 심각도 판단, 맥락 설명)
```

### 서비스별 역할 분담 (S4 의견)

| 서비스 | 할 일 |
|--------|-------|
| **S2** | SAST 도구 실행 (Semgrep, CodeQL 등), 결과 수집, API 계약에 findings 필드 추가 |
| **S3** | findings를 활용하는 프롬프트 재설계, confidence 로직 조정 (SAST+LLM 일치 시 신뢰도 상승 등) |
| **S4** | 변경 없음 (프롬프트가 더 길어질 뿐, 262K 컨텍스트 + KV cache 여유 충분) |

### S3가 프롬프트에서 고려할 점

- SAST findings를 구조화된 형태로 프롬프트에 삽입 (JSON 또는 마크다운 테이블)
- LLM에게 "찾아라"가 아니라 "이 finding이 실제 위협인지 검증하고, 자동차 보안 맥락에서 심각도를 판단하라"로 지시 변경
- SAST가 놓친 부분도 LLM이 보완적으로 탐지하는 2차 패스 고려

## 제안 2: GraphRAG (우선순위 낮음, 참고용)

### 논의 결과

GraphRAG를 조사하면서 두 가지 별개의 문제를 구분했습니다:

#### A. 위협 지식 그래프 — 지금은 불필요

현재 Qdrant 벡터 DB의 위협 지식 규모에서는 벡터 검색으로 충분합니다. 지식 그래프가 빛나려면:
- 수만 건 이상의 지식 베이스
- 다중 홉 추론 필요 (CWE 계열 → 공격 패턴 → 자동차 도메인 완화 전략)
- 글로벌 질의 필요

**현 단계에서는 위협 지식의 양과 품질을 늘리는 게 그래프보다 ROI가 높습니다.**

#### B. 코드 지식 그래프 — 에이전트 단계에서

코드의 AST, Call Graph, Data Flow Graph 등은 사실 SAST 도구가 이미 만드는 것입니다. 이걸 "GraphRAG"로 RAG 파이프라인에 넣는 게 아니라, **에이전트의 도구(tool)로 노출**하는 게 맞습니다:

```
LLM agent: "이 함수의 caller를 알고 싶다" → tool_call: code_graph.get_callers("process_can_msg")
LLM agent: "이 변수의 taint를 추적하고 싶다" → tool_call: code_graph.trace_taint("input_buf")
```

이건 에이전트 아키텍처 전환 시 설계할 문제이므로 지금 착수할 필요는 없습니다.

## S4 영향도

S4는 이 모든 단계에서 **변경 불필요**합니다. 프롬프트 내용이 달라질 뿐 추론 자체는 동일하며, 토큰 증가에 대한 여유도 충분합니다 (KV cache 2.1%, 262K 컨텍스트).

다만 SAST findings가 포함되면 프롬프트 토큰 수가 증가할 수 있으므로, 필요 시 배치 처리 성능 재측정을 지원하겠습니다.
